Contribuir a Qiskit¶

Qiskit es un proyecto de código abierto comprometido a hacer la computación cuántica disponible a personas de cualquier perfil de conocimiento. Esta página describe cómo puedes unirte a la comunidad de Qiskit para lograr este objetivo.

Dónde encontrar el código¶

El código Qiskit se ubica en el repositorio GitHub de Qiskit, donde puedes encontrar los proyectos individuales que conforman Qiskit, incluyendo

  • Qiskit Terra

  • Qiskit Aer

  • Qiskit Ignis

  • Qiskit Aqua

  • Qiskit IBMQ Provider

  • Tutoriales de Qiskit

  • Documentación de la API de Qiskit

Primeros pasos¶

Aprende cómo los miembros de la comunidad de Qiskit

  • Se relacionan entre sí

  • Discuten ideas

  • Buscan ayuda cuando se atoran

  • Se mantienen informados con noticias de la comunidad

  • Matienen un estilo consistente

  • Construyen paquetes de Qiskit desde el código fuente

Código de Contribución¶

Guía de estilo¶

Empleamos la herramienta Pylint para mantener un estilo de código consistente en el proyecto y pycodesytle para verificar que las contribuciones de código se ajustan y respetan la guía de estilo del proyecto. Para verificar que tus modificaciones se ajustan a la guía de estilo ejecuta: tox -elint

Acuerdo de Licencia del Contribuidor¶

Antes de que puedas enviar cualquier contribución a la comunidad, es necesario que firmes un acuerdo de licencia del contribuidor (conocido por sus siglas en inglés, CLA). Al firmar el CLA, afirmas ser el autor de la contribución y que tu contribución se ajusta a los términos de la licencia Apache-2.0.

Cuando contribuyes al proyecto de Qiskit con un pull request, un bot evaluará si has firmado el CLA. Si se requiere, el bot comentará sobre el pull request, incluyendo una liga para aceptar el acuerdo. El CLA individual está disponible para revisar como PDF.

Nota

Si tu contribución es parte de tu empleo o tu contribución es propiedad de tu empleador, entonces seguramente necesitas firmar también un CLA corporativo y enviarlo al correo <qiskit@us.ibm.com>.

Pull Requests¶

Utilizamos pull requests de Github para aceptar las contribuciones.

Aunque no es obligatorio, abrir un nuevo asunto sobre un error que estás tratando de arreglar o una nueva funcionalidad en la que estás trabajando antes de hacer un pull request es importante, de cara a poder discutir con la comunidad de Qiskit tu trabajo. Este asunto genera un espacio de diálogo para hablar sobre la idea y sobre cómo la comunidad puede trabajar conjuntamente para implementarlo en código. Además, esto permite conocer a la comunidad que estás trabajando en un asunto particular y, si necesitas ayuda, siempre puedes hacer referencia al asunto abierto para discutir la idea con otros miembros del equipo y la comunidad.

Si has escrito algo de código pero necesitas ayuda, quieres obtener retroalimentación antes de terminarlo, o quieres compartirlo y discutir antes de finalizar la implementación, puedes abrir un Pull Request de tipo Work in Progress. Cuando crees el PR, inicia el título con la etiqueta [WIP] (para Work in Progress). Quien revise el código podrá ver que el PR no está en su estado final y cambiará. También significa que no se integrará el código hasta que esté listo. Tú o un supervisor pueden eliminar la etiqueta [WIP] cuando el código esté listo para ser revisado completamente para su integración.

Mensajes de commit¶

El contenido del mensaje de confirmación que describe un cambio es tan importante como el propio cambio. El mensaje proporciona el contexto no sólo para la revisión del código, sino también el historial de cambios en el registro de Git. Un mensaje detallado hará que sea más fácil revisar tu código, y también proporcionará contexto cuando alguien revise el cambio en el futuro. Cuando escribas un mensaje de confirmación, recuerda estos detalles importantes:

No supongas que quien revisa entiende cual era el problema original.

When reading an issue, after a number of back & forth comments, it is often clear what the root cause problem is. The commit message should have a clear statement as to what the original problem is. The bug is merely interesting historical background on how the problem was identified. It should be possible to review a proposed patch for correctness from the commit message, without needing to read the bug ticket.

No suponga que el código es obvio o auto-documentado.

Lo que es evidente para una persona, podría no serlo para otra. Siempre documenta cuál era el problema original y cómo se está arreglando. Esto aplica para todos los cambios excepto errores de dedo o modificaciones al espacio en blanco.

Describe por qué se está realizando un cambio.

Un error común es sólo documentar cómo se ha escrito el código, sin describir * por qué * el desarrollador eligió hacerlo de esa manera. Se debe describir la estructura general del código, particularmente para los cambios grandes, pero lo más importante es describir la intención/motivación detrás de los cambios.

Lee el mensaje de confirmación para ver si sugiere una estructura de código mejorada.

A veces, al revisar un mensaje de confirmación grande, resulta obvio que el cambio debería haberse dividido en dos o más partes. No tengas miedo de volver atrás y dividir el cambio en Pull Request diferentes.

Asegúrese de que la información suficiente para decidir si se debe revisar.

Cuando GitHub envía un correo electrónico alertando de una nueva solicitud de Pull Request, solamente se incluye un mínimo de información - usualmente sólo el mensaje de commit y la lista de cambios en los archivos. Debido al alto volumen de parches, un mensaje de commit debe contener suficiente información para que quien la revise pueda encontrar el parche a revisar.

La primera linea del mensaje de commit es la más importante.

En los commits de Git, la primera línea del mensaje del commit tiene particular importancia. Esta es usada por defecto como el título de la Pull Request, asunto de la notificación por correo electrónico, mensajes de anotación de git, anotación del visualizador gitk, mensajes de commit para merge y muchos otros lugares donde el espacio es limitado. Además de resumir el cambio, debe explicar detalladamente qué parte del código es afectada por este.

Además, la primera línea del mensaje de commit se convierte en una entrada en el registro de cambios si el PR está etiquetado para incluirse en el changelog. Es crítico que escribas un resumen claro y conciso.

Describa las limitaciones del código actual.

Si el código que está siendo modificado aún tiene potencial para mejoras posteriores, o alguna limitación, menciónalo en el mensaje de commit. Esto demuestra a quien revisa que se ha considerado un panorama más amplio, y qué compensaciones se han echo en cuanto a objetivos de corto plazo contra deseos de largo plazo.

Incluye referencias a los problemas.

Si las modificaciones del commit están relacionadas con algún problema, asegúrate de anotarlo en el mensaje del commit. Usa la sintaxis:

Fixes #1234

si soluciona el problema (GitHub cerrará el problema cuando se haga merge del Pull Request).

La principal regla a seguir es:

El mensaje de commit debe contener toda la información necesaria para entender en su totalidad y revisar si el parche es apropiado. Menos no es más.

Documentando Tu Código¶

Si modificas un elemento, asegurate de actualizar las docstrings asociadas al igual que las partes de la documentación ubicadas en el folder docs/apidocs del repositorio correspondiente. Para compilar localmente la documentación específica de un elemento, ejecuta tox -edocs, esto compila localmente la documentación y guarda el resultado en docs/_build/html. Además, la tarea Docs CI en el procesamiento segmentado de Azure ejecutará esto y generará un archivo zip con la salida, el cual puedes descargar y ver de manera local.

Si tienes algún problema con la documentación común que se encuentra en el repositorio Qiskit/qiskit, puedes abrir un documentation issue si encuentras algún error, tienes alguna nueva funcionalidad por documentar o crees que se puede agregar material a los documentos existentes.

Primeras Contribuciones Recomendadas¶

Si quieres contribuir a Qiskit pero no estás seguro de cómo empezar a hacerlo, la etiqueta de ``good first issue``en la lista de problemas de un proyecto destaca aquellos problemas apropiados para quienes se inician en el proyecto. Otros contribuidores han revisado y marcados estos problemas como algo en lo que un nuevo contribuidor puede trabajar sin problemas. En otras palabras, no se requiere un conocimiento profundo de Qiskit para solucionar estos problemas.

Política de Obsolesencia¶

Los usuarios de Qiskit necesitan saber si alguna funcionalidad o una API que utilizan se mantendrá habilitada por el software el día de mañana. Para el usuario es importante saber bajo que condiciones el proyecto puede eliminar (o modificar de forma irreversible) una funcionalidad o API. Para gestionar anticipaciones, la siguiente política describe la forma en la que Qiskit maneja la obsolesencia/eliminación de una API o funcionalidad:

1. Features, APIs, or configuration options are marked deprecated in the code. Appropriate DeprecationWarning class warnings will be sent to the user. The deprecated code will be frozen and receive only minimal maintenance (just so that it continues to work as-is).

2. A migration path will be documented for current users of the feature. This will be outlined in the both the release notes adding the deprecation, and the release notes removing the feature at the completion of the deprecation cycle. If feasible, the warning message will also include the migration path. A migration path might be «stop using that feature», but in such cases it is necessary to first judge how widely used and/or important the feature is to users, in order to determine a reasonable obsolescence date.

3. An obsolescence date for the feature will be set. The feature must remain intact and working (although with the proper warning being emitted) in all releases pushed until after that obsolescence date. At the very minimum, the feature (or API, or configuration option) should be marked as deprecated (and continue to be supported) for at least three months of linear time from the release date of the first release to include the deprecation warning. For example, if a feature were deprecated in the 0.9.0 release of Terra, which was released on August 22, 2019, then that feature should still appear in all releases until at least November 22, 2019. Since releases do not occur at fixed time intervals, a deprecation warning may only occur in one release prior to removal.

Es importante destacar que este plazo es mínimo. Para funcionalidades importantes es recomendable que la funcionalidad obsoleta se mantenga habilitada al menos el doble de tiempo. Además, gracias a la política de ramificación estable, la eliminación de funcionalidades obsoletas sólo se puede producir durante el lanzamiento de versiones menores; no son apropiadas para tenerlas en una nueva versión.

Advertencias de Obsolesencia¶

La manera apropiada de crear una advertencia de obsolesencia es utilizando la función warn``del `modulo warnings <https://docs.python.org/3/library/warnings.html>`__ en la biblioteca estándar de Python. La categoría de advertencia debe ser de la clase ``Deprecation Warning. Un ejemplo de esto sería:

import warnings

def foo(input):
    warnings.warn('The qiskit.foo() function is deprecated as of 0.9.0, and '
                  'will be removed no earlier than 3 months after that '
                  'release date. You should use the qiskit.bar() function '
                  'instead.', DeprecationWarning, stacklevel=2)

Una cosa a tener en cuenta es el argumento kwarg «stack_level» en la llamada de warn(). Este argumento es usado para especificar qué nivel en el stack de llamada será usado como la línea que inicia la advertencia. Normalmente, «stack_level» debería tener valor 2, ya que ésto mostrará la linea llamando al contexto donde apareció la advertencia. En el ejemplo que aparece arriba, sería el que llama a «foo()». Si no has establecido esto, la advertencia mostraría que fue causada por la línea en la función foo(), que no es muy útil para usuarios que intentan determinar el origen de la llamada obsoleta. Sin embargo, este valor puede ser ajustado, dependiendo del stack de llamada y de donde «warn()» sea llamado. Por ejemplo, si la advertencia siempre aparece por culpa de un método privado que solo es llamado en un lugar, «stack_level=3» podría ser adecuado.

Política de Ramificación Estable¶

La rama estable está destinada a ser una fuente segura de soluciones para errores de alto impacto y problemas de seguridad que se han solucionado en el master desde su lanzamiento. Al revisar un PR de rama estable, debemos equilibrar el riesgo de cualquier parche dado con el valor que proporcionará a los usuarios de la rama estable. Solo una clase limitada de cambios son apropiados para su inclusión en la rama estable. Un parche grande y arriesgado para un problema importante podría tener sentido, al igual que una solución trivial para un caso bastante oscuro de manejo de errores. Se debe tomar en cuenta un numero de factores cuanto se esté considerando un cambio:

  • El riesgo de la regresión: hasta los cambios más pequeños conllevan cierto riesgo de romper algo, por eso queremos evitar a toda costa cualquier regresión en la rama estable.

  • El beneficio de visibilidad para el usuario: ¿estamos arreglando algo que los usuarios podrían notar? y, de ser así, ¿qué tan importante es?

  • Cuán autónoma es la solución: si soluciona un problema importante, pero también transforma mucho contenido del código, probablemente valga la pena pensar en cómo podría ser una solución menos riesgosa.

  • Si el arreglo está ya en la rama master: un cambio debe ser un backport de un cambio ya fusionado en la master, a no ser que simplemente el cambio no tenga sentido en la master.

Procedimiento de backporting:¶

Al hacer backporting de un parche desde la master a la rama estable, queremos mantener una referencia del cambio en la master. Al crear la rama para el PR estable, utilice:

$ git cherry-pick -x $master_commit_id

No obstante, esto solo funciona para pequeños parches autocontenidos en el master. Si necesita hacer backport de un subconjunto de un commit mayor (de un PR aplastado, por ejemplo) en el master, hágalo manualmente. En estos casos, añada:

Backported from: #master pr number

de esta manera podemos rastrear el origen del subconjunto de cambios, incluso si un cherry-pick estricto no tiene sentido.

Si el parche no está proponiendo un cherry-pick limpio, puede ayudar resolviendo los conflictos usted mismo y proponiendo un parche resultante. Mantenga las líneas de conflictos en el mensaje de confirmación para ayudar a revisar el parche estable.

Etiquetas de backport¶

Bugs o PRs etiquetados con stable backport potential son bugs que también afectan a la versión estable y pueden ser adecuados para hacer backport sólo cuando hayan solucionados en el master. Una vez que se ha propuesto el backport, se debe eliminar la etiqueta.

Incluya``[Stable]``en el título de los PR hacia la rama estable, como señal de que no es un error haber seleccionado esa rama. También haga referencia al número del PR de la rama master del que se está transfiriendo.

Contribuir a la Documentación¶

La documentación de Qiskit sigue la filosofía documentación como código, principalmente recogida del código de Qiskit usando el estilo de documentación para APIs.

La documentación se construye desde la rama master de Qiskit/qiskit/docs usando Sphinx. La mayoría de la documentación, bajo Referencias de la API, se toma de los comentarios del código en los repositorios indicados en Dónde encontrar el código.

Estructura de la Documentación¶

La documentación de Qiskit está estructurada para tener la mayor cantidad posible de documentación en como docstring. Esto facilita adiciones y correcciones durante el desarrollo, ya que la mayoría de la documentación se encuentra en el código que se está cambiando. Hay tres niveles en la estructura de documentación normal en Terra:

Los archivos .rst en docs/apidocs

Estos archivos son utilizados por Sphinx para seleccionar qué módulos incluir en la documentación generada. Esta contiene dos partes: una internal reference (referencia interna) o una cross reference (referencia cruzada respecto al módulo), que pueden ser utilizadas para enlaces internos dentro de la documentación, y una automodule directive (directiva de automódulo) utilizada para analizar el docstring del módulo desde una ruta de importación especificada. Por ejemplo, el archivo dagcircuit.rst contiene:

.. _qiskit-dagcircuit:


.. automodule:: qiskit.dagcircuit
   :no-members:
   :no-inherited-members:
   :no-special-members:

El único archivo .rst fuera de este es qiskit.rst, que contiene las tablas de contenido. Si estás añadiendo un nuevo archivo .rst para un módulo nuevo de la documentación, asegúrate de añadirlo también al toctree en ese archivo.

Docstring a nivel de módulo

Esta docstring está en el nivel de módulo para el módulo especificado en la carpeta `` automodule`` en el archivo rst. Si el módulo especificado es un directorio / namespace, la docstring se debe especificarse en el archivo __init__.py para ese directorio. Esta docstring a nivel de módulo contiene más detalles sobre el módulo que se está documentando. La estructura normal de esta docstring es describir todas las clases y funciones de la API pública contenida en ese módulo. Esto normalmente se hace usando la directiva autosummary (o directivas autodoc <http://www.sphinx-doc.org/en/master/usage/extensions/autodoc.html> __ directamente si el módulo es simple, como en el caso de `` qiskit.execute``). La directiva autosummary se usa para auto-documentar una lista de diferentes elementos de Python (clases, funciones, etc.) directamente sin tener e invocar manualmente las directivas de auto-documentación para cada uno. La docstring de nivel de módulo es donde puede proporcionar una descripción general de alto nivel de la funcionalidad que proporciona el módulo. Esto normalmente se hace agrupando los diferentes componentes de la API pública en múltiples subsecciones.

Por ejemplo, como en el ejemplo anterior del módulo dagcircuit, los contenidos de la docstring del módulo para qiskit/dagcircuit/__init__.py serían:

"""
=======================================
DAG Circuits (:mod:`qiskit.dagcircuit`)
=======================================
.. currentmodule:: qiskit.dagcircuit
DAG Circuits
============
.. autosummary::
   :toctree: ../stubs/
   DAGCircuit
   DAGNode
Exceptions
==========
.. autosummary::
   :toctree: ../stubs/
   DAGCircuitError
"""

Nota

Esto es solamente un ejemplo y la docstring real para el módulo dagcircuit podría diferir.

La docstring real para los elementos listados en la docstring del módulo

Debería esforzarse en documentar detalladamente todas las interfaces públicas expuestas usando ejemplos cuando sea necesario. Para docstrings se usa Google Python Style Docstrings. Esto se analiza usando la extensión `napoleon sphinx<https://www.sphinx-doc.org/en/master/usage/extensions/napoleon.html>`__. La documentación de napoleon contiene un buen ejemplo sobre cómo debe de ser el formato de la docstring.

Nota

Puedes usar cualquier directiva de Sphinx o del formato rst en una docstring mientras sea razonable. Por ejemplo, una extensión comúnmente usada es la directiva jupyter-execute, que se utiliza para ejecutar un bloque de código en Jupyter y mostrar tanto el código como su salida. Esto es particularmente útil para las visualizaciones.

Integración de la Documentación¶

La documentación alojada en https://qiskit.org/documentation/ cubre el proyecto Qiskit por completo; Terra es solamente un componente de este. Como tal, la documentación generada para la versión alojada está construida usando el repositorio de meta paquetes de Qiskit https://github.com/Qiskit/qiskit. Cuando los commits se mezclan en el repositorio, la salida de Sphinx generada se sube al sitio web qiskit.org. La salida generada por Sphinx está configurada para encajar con la documentación de las versiones de los elementos de Qiskit instalados en el meta-paquete en ese momento. Por ejemplo, si la versión del meta-paquete es la 0.13.0, entonces se copiará la documentación de la versión 0.10.0 de Terra. Cuando los requisitos del meta-paquete son modificados, entonces empezará a extraer documentación de la nueva versión. Esto significa que se necesitarán arreglos para la documentación incorrecta de la API en la nueva versión. Los arreglos a la documentación son backports válidos para un versión estable según la política de la rama estable (vea esta sección abajo).

Durante el proceso de generación, los contenidos de cada elemento de docs/apidocs/ son recursivamente duplicados en una copia compartida de doc/apidocs/ en el repositorio del meta paquete junto con todos los otros elementos. Esto significa que lo que haya en esta raíz docs/apidocs de cada elemento en una versión acabará estando en la raíz de https://qiskit.org/documentation/apidoc/.

Traduciendo la Documentación¶

La documentación de Qiskit se traduce (localiza) usando Crowdin, una plataforma web para localización de software que permite organizar y coordinar proyectos de traducción así como colaborar con comunidades para traducir materiales. Crowdin permite que nuestra comunidad de traductores amplifique su impacto reutilizando automáticamente el trabajo invertido en la traducción de una oración para traducir oraciones similares. Crowdin también hace que las traducciones sean resilientes a diferentes tipos de cambios del material original, como mover las oraciones de lugar, incluso entre archivos.

Las solicitudes de localización de Qiskit se gestionan en el repositorio Qiskit Translations. Para contribuir a la localización de Qiskit, siga estos pasos:

  1. Añada su nombre (o ID) al archivo LOCALIZATION_CONTRIBUTORS.

  2. Realice un pull request (PR) para incluir el cambio. Asegúrese de seguir la plantilla para abrir una pull request.

    Nota

    • Cada colaborador debe crear su propio PR y firmar el acuerdo de licencia de colaborador (CLA).

    • Por favor, indique en el resumen del PR el idioma al que desea contribuir.

    • Si tienes un issue abierto en el que se solicita un idioma, agrega el link del issue al Pull Request.

  3. Se le pedirá que firme el Acuerdo de Licencia de Contribuidor (CLA); por favor, hágalo.

  4. Un mínimo de 3 contribuidores por idioma es necesario, para que el idioma sea agregado y reciba el soporte oficial de los administradores del proyecto de localización.

  5. Entre el grupo de contribuyentes, se debe identificar un líder de traducción para que sirva de enlace con los administradores del proyecto de localización. El líder debe contactar a: Yuri Kobayashi (yurik@jp.ibm.com) por correo electrónico.

  6. En el proyecto Qiskit-Docs Crowdin, elija el idioma al que desea contribuir.

    Nota

    Como es mencionado en la publicación del post, “Qiskit in my language is Qiskit, debemos asegurarnos que los idiomas traducidos tienen la comunidad suficiente de soporte para construir un equipo de traducción, correctores, y líderes de traducción. Si desea ser un líder de traducción o está interesado en unirse a un nuevo equipo de proyecto de traducción, puede abrir un GitHub issue para empezar un debate con el equipo de Qiskit y reclutar miembros del proyecto de traducción.

  7. Haz click en el botón Unirse (Join) y pega la URL de tu PR en el cuadro de diálogo en el que se te pregunta por qué quieres unirte al proyecto de Crowdin.

Los administradores del proyecto Crowdin revisarán su solicitud y le darán acceso tan pronto como sea posible.

Construyendo con Código Fuente¶

Puedes construir una copia local de la documentación desde tu clonación local del repositorio Qiskit/qiskit de la siguiente manera:

  1. Clonar el repositorio de Qiskit.

    git clone https://github.com/Qiskit/qiskit.git

  2. La clonación crea un folder local llamado qiskit.

    cd qiskit

  3. Genere la documentación navegando a tu copia local de Qiskit/qiskit y ejecutando el siguiente comando en una ventana de terminal.

    tox -edocs

    Si todavía no tienes instalado el comando tox, instálalo ejecutando:

    pip install tox

Mientras haces cambios a tus archivos RST locales, puedes actualizar tus archivos HTML navegando a /doc/ y ejecutando lo siguiente en una terminal:

tox -edocs

Esto construirá una nueva versión de HTML con estilos de tu documentación local en el subdirectorio /docs/_build/html/.

Instalando con Código Fuente¶

Instalando los elementos usando el código fuente te permite acceder a la última versión de Qiskit en lugar de la versión del repositorio de Python Package Index (PyPl). Esto te da la capacidad de inspeccionar y extender la última versión del código de Qiskit más eficientemente.

Al instalar los elementos y componentes a partir de la fuente, por defecto se usará su versión marcada como development (que corresponde a la rama git master), y no la versión stable (que contiene la misma base de código que la publicada en los paquetes pip). Dado que la versiones development de cada elemento o componente normalmente incluyen nuevas funcionalidades y cambios, generalmente será necesario también instalar las versiones development del resto del módulo.

Nota

Los paquetes de Terra y Aer requieren un compilador para construirse a partir del código fuente antes de poder instalarlos. Ignis, Aqua y el backend de IBM Quantum Provider no necesitan un compilador.

La instalación de los elementos a partir del código fuente requiere que se respete el siguiente orden de instalación para evitar que se instalen versiones inferiores de elementos si la versión en pip es inferior a la de las versiones fuentes:

  1. qiskit-terra

  2. qiskit-aer

  3. qiskit-ignis

  4. qiskit-aqua

  5. qiskit-ibmq-provider (si quieres conectarte a los dispositivos de IBM Quantum o al simulador online)

Para trabajar en varios components y elementos simultáneamente, sigue los siguientes pasos para cada elemento.

Nota

Debido al uso de namespaces en los paquetes de Python, se debe tener cuidado al instalar los paquetes. Si planea instalar cualquier elemento desde la fuente, no use el meta paquete `` qiskit``. Además, siga esta guía y use un entorno virtual separado para el desarrollo. Si elige mezclar una instalación existente con su desarrollo, consulte https://github.com/pypa/sample-namespace-packages/blob/master/table.md para conocer las combinaciones de métodos de instalación que funcionan juntas sin problema.

Configurar el Ambiente de Desarrollo Virtual¶

conda create -y -n QiskitDevenv python=3
conda activate QiskitDevenv

Instalar Terra a partir del código fuente¶

Para instalar a partir del código fuente es necesario que tenga un compilador de C++ en tu ordenador que soporte C++11.

En la mayoría de las plataformas Linux, el compilador necesario GCC está ya instalado.

Si utilizas macOS, puedes instalar el compilador Clang al instalar XCode. Comprueba si tienes XCode y Clang instalados abriendo una ventana de terminal y ejecutando los siguientes comandos.

clang --version

Instala XCode y Clang usando los siguientes comandos.

xcode-select --install

En Windows, lo más fácil de instalar es el compilador Visual C++ desde Build Tools for Visual Studio 2017. Por otro lado puedes instalar Visual Studio versión 2015 o 2017, asegurándote de seleccionar las opciones para instalar el compilador C++.

Una vez que los compiladores estén instalados, todo está listo para instalar Qiskit Terra.

  1. Clonar el repositorio de Terra.

    git clone https://github.com/Qiskit/qiskit-terra.git

  2. La clonación del repositorio crea un folder local llamado qiskit-terra.

    cd qiskit-terra

  3. Instalar los requerimientos de librerías de Python desde tu directorio qiskit-terra.

    pip install cython

  4. Si quieres ejecutar pruebas o verificaciones de lint, instala los requerimientos de desarrollador.

    pip install -r requirements-dev.txt

  5. Instalar qiskit-terra.

    pip install .

Si desea instalarlo en modo editable, es decir que los cambios en el código del proyecto no requieran ser instalados de nuevo, puede hacerlo con:

pip install -e .

Puede ejecutar los ejemplos de código después de instalar Terra. Puede ejecutar el ejemplo con el siguiente comando.

python examples/python/using_qiskit_terra_level_0.py

Nota

Si no planea instalar algún otro componente, qiskit-terra emitirá un RuntimeWarning advirtiendo que qiskit-aer y qiskit-ibmq-provider no están instalados. Esto se hace porque los usuarios suelen utilizar ambos elementos adicionales, pero no se dan cuenta de que no están instalados o de que la instalación Aer o IBM Quantum Provider no se realizó con éxito por alguna razon. Si quiere desactivar estas advertencias, añada:

import warnings
warnings.filterwarnings('ignore', category=RuntimeWarning,
                        module='qiskit')

antes de cualquier importación de paquetes qiskit en tu código. Eso suprimirá la advertencia sobre los componentes qiskit-aer y qiskit-ibmq-provider faltantes, pero seguirán mostrándose otras advertencias de qiskit u otros paquetes.

Instalar Aer a partir del código fuente¶

  1. Clona el repositorio Aer.

    git clone https://github.com/Qiskit/qiskit-aer

  2. Instalar los requerimientos del construcción.

    pip install cmake scikit-build cython

Después de esto, los pasos para instalar Aer dependerán del sistema operativo que estés usando. Dado que Aer es un programa compilado en C++ con una interfaz de Python, hay dependencias necesarias para construir Aer que no son de Python y que no se pueden instalar de manera universal en todos los sistemas operativos.

  1. Instalar requerimientos del compilador.

    Construir Aer requiere un compilador de C++ y las cabeceras de desarrollo.

    Si estas usando Fedora o una distribución de Linux equivalente, instala usando:

    dnf install @development-tools

    Para Ubuntu/Debian instala usando:

    apt-get install build-essential

  2. Instala las cabeceras de desarrollo OpenBLAS.

    Si estas usando Fedora o una distribución de Linux equivalente, instala usando:

    dnf install openblas-devel

    Para Ubuntu/Debian instala usando:

    apt-get install libopenblas-dev

  1. Instalar dependencias.

    Para usar el compilador Clang en macOS, necesitas instalar una librería adicional para soportar OpenMP. Puedes usar brew para instalar esta y otras dependencias.

    brew install libomp

  2. En seguida, instala una implementación de BLAS; OpenBLAS es la opción por defecto.

    brew install openblas

    Después, instala Xcode Command Line Tools.

    xcode-select --install

En Windows debes usar Anaconda3 o Miniconda3 para instalar todas las dependencias.

  1. Instalar requerimientos del compilador.

    conda install --update-deps vs2017_win-64 vs2017_win-32 msvc_runtime

  2. Instalación de las dependencias binarias y de construcción.

    conda install --update-deps -c conda-forge -y openblas cmake

  1. Construir e instalar qiskit-aer directamente

    Si tiene pip<19.0.0 instalado y tu ambiente no requiere una compilación personalizada, ejecuta:

    cd qiskit-aer
pip install .

    Esto ejecutará la construcción de binarios e instalará Aer.

    De manera alternativa, si tiene instalado un pip más reciente, o tienes algún requerimiento específico, puedes hacer un paquete Python Wheel manualmente.

    cd qiskit-aer
python ./setup.py bdist_wheel

    Si necesita establecer una opción personalizada durante la construcción del paquete wheel, consulte Opciones personalizadas durante la construcción del wheel.

    Después de construir el Python wheel, éste será guardado en la carpeta dist/ del repositorio Aer. La versión exacta dependerá

    cd dist
pip install qiskit_aer-*.whl

    El nombre exacto del archivo wheel de salida depende de la versión actual de Aer que en desarrollo.

Opciones personalizadas durante la construcción del wheel¶

El sistema de compilación Aer utiliza scikit-build para ejecutar la compilación cuando se construye usando la interfaz de Python. Esta actúa como interfaz para que las setuptools hagan una llamada a CMake y compilen los binarios para tu sistema local.

Debido a la complejidad de la compilación de binarios es posible que tenga que pasar opciones a ciertas partes del proceso de construcción. La forma de pasar variables es:

python setup.py bdist_wheel [skbuild_opts] [-- [cmake_opts] [-- build_tool_opts]]

donde los elementos dentro de corchetes [] son opcionales, y skbuild_opts, cmake_opts, build_tool_opts deben ser reemplazados por banderas de tu elección. Una lista de opciones de CMake está disponible aquí: https://cmake.org/cmake/help/v3.6/manual/cmake.1.html#options. Por ejemplo, puede ejecutar algo como esto:

python setup.py bdist_wheel -- -- -j8

Esto es pasar el flag -j8`al sistema de compilación base (que en este caso es `Automake), indicando que quieres compilar en paralelo utilizando 8 procesos.

Por ejemplo, un caso de uso común de estas banderas (flags) en Linux es para indicar una versión específica del compilador de C++ a usar (normalmente se utiliza si el compilador por defecto es uno antiguo):

python setup.py bdist_wheel -- -DCMAKE_CXX_COMPILER=g++-7

el cuál comunicará a CMake que use el comando g++-7 en vez del predeterminado g++ al compliar Aer.

Otro uso común para esto, dependiendo de tu entorno, es para cuando debas especificar el nombre de la plataforma y apagar las ligas estáticas.

python setup.py bdist_wheel --plat-name macosx-10.9-x86_64 \
-- -DSTATIC_LINKING=False -- -j8

Aquí --plat-name es la bandera para las setuptools que especifica el nombre de plataforma a utilizar en los metadatos del paquete, -DSTATIC_LINKING es la bandera para hacer que CMake deshabilite el enlazado estático, y -j8 es la bandera para hacer que Automake utilice 8 procesos para la compilación.

Una lista de opciones comunes dependiendo de la plataforma es:

Plataforma

Herramienta

Opción

Caso de uso

Todas

Automake

-j

Seguido de un número, indica la cantidad de procesos a utilizar para la compilación.

Linux

CMake

-DCMAKE_CXX_COMPILER

Se utiliza para especificar un compilador específico de C++; esto se necesita a menudo si el g++ que tienes es antiguo.

OSX

setuptools

–plat-name

Usado para especificar el nombre de la plataforma en el paquete de salida de Python.

OSX

CMake

-DSTATIC_LINKING

Utilizado para especificar si se usará enlazado estático o no.

Nota

Algunas de estas opciones no son específicas de la plataforma. Estas plataformas particulares están enumeradas porque son comúnmente usadas en el entorno. Consulte la documentación de la herramienta para obtener más información.
